Google Summer of Code 2025: AnkiDroid Multi-Profile Support

1) Project Information

Project Title: AnkiDroid Multi-Profile Implementation: Bringing User Profiles to Android

Project Prerequisites: - AnkiDroid GitHub Repository - Multi-Profile Feature Request #2545 - Anki Desktop Profile Implementation

Project Summary: This project addresses a critical feature gap in AnkiDroid: the lack of multi-profile support, which has been requested by users for over 5 years. Currently, AnkiDroid only supports a single profile per installation, forcing users with multiple study domains or shared devices to maintain separate installations or constantly re-sync accounts. I will implement a comprehensive multi-profile system that enables users to create, switch between, and manage multiple profiles on a single device, each with isolated collections, preferences, and AnkiWeb sync credentials. The implementation will follow the proven architecture from Anki Desktop while adapting to Android’s storage framework and security model. Key deliverables include a secure profile management interface, optimized storage structure for multiple collections, AnkiWeb integration with per-profile credentials, and a seamless migration path for existing users. This feature will significantly improve AnkiDroid’s usability in family, educational, and multi-domain learning contexts.

5) Detailed Project Understanding

Current Limitations in AnkiDroid

AnkiDroid currently supports only a single profile per installation, which creates several significant limitations:

  1. Shared Device Problems:
  • Families sharing tablets cannot maintain separate progress
  • Educational settings cannot support multiple students
  • User privacy is compromised on shared devices
  1. Multiple Domain Challenges:
  • Users studying multiple subjects cannot separate their collections
  • AnkiWeb sync requires complete account switching
  • Statistics and progress are merged across all subjects
  1. Technical Implications:
  • Single collection database limits architectural options
  • Hard-coded paths for media storage
  • Global preferences instead of per-profile settings

The lack of multi-profile support is a top-requested feature (Issue #2545) with 100+ comments and reactions, demonstrating significant user demand.

Detailed Architecture Proposal

My implementation will address these limitations through a comprehensive architecture:

  1. Profile Management Layer:
  • ProfileManager singleton to handle profile operations
  • Secure storage of profile metadata and credentials
  • Profile switching with proper lifecycle management
  1. Storage Architecture:
  • Profile-specific directories for collections and media
  • Isolated SQLite databases for each profile
  • Shared code that operates on the “active” profile
  1. Security Implementation:
  • Encrypted storage of AnkiWeb credentials
  • Optional PIN/biometric protection for profiles
  • Secure export/import of profile data
  1. Integration Points:
  • Modified collection loader to use active profile path
  • Updated sync client to use profile-specific credentials
  • Adapted preferences system for per-profile settings

C) Profile Manager Implementation

/**
* Singleton manager for AnkiDroid profiles
*/
object ProfileManager {
   private const val PROFILES_FILE = "profiles.json"
   private var activeProfile: Profile? = null
   private var profiles = mutableListOf<Profile>()
   
   /**
    * Load profiles from storage
    */
   fun loadProfiles(context: Context) {
       // Load profiles from encrypted storage
       val encryptedPrefs = createEncryptedPrefs(context, PROFILES_FILE)
       val profilesJson = encryptedPrefs.getString("profiles", "[]")
       profiles = Json.decodeFromString(profilesJson)
   }
   
   /**
    * Create a new profile
    */
   fun createProfile(name: String, colorIndex: Int, iconIndex: Int): Profile {
       val profile = Profile(
           name = name,
           colorIndex = colorIndex,
           iconIndex = iconIndex
       )
       profiles.add(profile)
       saveProfiles()
       return profile
   }
   
   /**
    * Switch to a different profile
    */
   suspend fun switchToProfile(context: Context, profileId: String): Boolean {
       // Verify profile exists
       val profile = profiles.find { it.id == profileId } ?: return false
       
       // Close current collection if open
       CollectionManager.closeCollection()
       
       // Update active profile
       activeProfile = profile
       profile.lastAccessed = System.currentTimeMillis()
       saveProfiles()
       
       // Update collection path
       val profilePath = getProfilePath(context, profile.id)
       CollectionManager.setCollectionPath(profilePath)
       
       // Load preferences for this profile
       PreferenceManager.loadProfilePreferences(context, profile.id)
       
       return true
   }
   
   /**
    * Get path for profile collection
    */
   fun getProfilePath(context: Context, profileId: String): String {
       return context.getExternalFilesDir(null)?.absolutePath + 
              "/profiles/" + profileId + "/collection.anki2"
   }
   
   // Additional methods for profile management...
}

D) Integration with AnkiWeb Sync

/**
* Extension of existing sync client to support multiple profiles
*/
class ProfileAwareHttpSyncer(private val profileId: String) : HttpSyncer {
   
   override fun sync(): SyncResult {
       // Get credentials for specific profile
       val profile = ProfileManager.getProfile(profileId)
       val username = profile.syncUsername
       val passwordHash = profile.syncPasswordHash
       
       if (username.isNullOrEmpty() || passwordHash.isNullOrEmpty()) {
           return SyncResult.NO_CREDENTIALS
       }
       
       // Set up sync with profile-specific credentials
       setupSync(username, passwordHash)
       
       // Proceed with normal sync process
       return super.sync()
   }

}
